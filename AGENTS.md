# AGENTS.md

## Purpose and scope

This file is the working guide for agents and contributors operating anywhere in this repository. It describes the repository as it exists, not only the intended final system.

The project researches computer-vision methods for detecting water in camera images and ultimately detecting overflow around phosphate-washery basins. The immediate technical goal is accurate binary semantic segmentation (`water` versus `background`). Overflow detection is a later stage built from a reliable segmentation mask, camera-specific geometry, and temporal confirmation.

Read `PLAN.md` before making architectural or experimental changes. It is the most complete statement of the research strategy. `README.md` is the project overview and introductory explanation. When documentation disagrees with executable code, verify the current code and data before changing behavior.

## Current maturity

This is a research prototype, mostly implemented through notebooks.

- The reusable Python package contains data-loading, threshold-baseline, metric, visualization, and rule-based overflow helpers.
- The OpenCV/Tkinter GUI demonstrates only the color-threshold baseline.
- The trained U-Net and most of its training/evaluation code live in a notebook, not in reusable package modules.
- The overflow notebook still uses an all-zero placeholder mask; it does not run the U-Net checkpoint.
- `main.py` is the original placeholder and only prints a greeting.
- There is no command-line interface, inference service, automated test suite, CI workflow, packaging configuration, common training script, or common evaluation script yet.
- The repository is not a production overflow alarm. The current overflow score is a heuristic, not a calibrated probability.

Do not describe planned components as already implemented.

## Core data contracts

Preserve these contracts unless the change is explicitly a breaking redesign.

### Images

- `load_image()` opens with Pillow, converts to RGB, optionally resizes with bilinear interpolation, and returns an HWC `float32` NumPy array normalized to `[0, 1]`.
- Pillow `size` values are `(width, height)`, while NumPy shapes are `(height, width, channels)`.
- OpenCV conversions must explicitly use RGB-aware constants such as `cv2.COLOR_RGB2HSV`; Pillow-loaded data is not BGR.

### Masks

- `load_mask()` converts to grayscale and returns a 2-D `uint8` mask containing `0` and `1`.
- Mask resizing must always use nearest-neighbor interpolation.
- `load_mask(threshold=...)` compares raw grayscale values, normally in `[0, 255]`.
- Metric functions use probability thresholds normally in `[0, 1]`. Do not confuse the loader threshold with the metric/prediction threshold.
- Preserve alignment by applying every geometric transformation identically to the image and mask.

### Pairing

`find_water_v2_pairs()` walks `water_v2/JPEGImages`, preserves each relative path under `water_v2/Annotations`, and looks for a mask with `.png`, `.jpg`, or `.jpeg`. It returns only images that have a corresponding mask. Sorted discovery order is intentional for reproducibility.

### Resizing and padding

- `load_pair()` resizes directly and can distort aspect ratio.
- `load_pair_with_padding()` letterboxes an image/mask pair, uses bilinear interpolation for the image and nearest-neighbor for the mask, and can return restoration metadata.
- `restore_mask_from_padding()` removes letterbox padding, binarizes the cropped mask, and restores the original resolution with nearest-neighbor interpolation.
- New segmentation work should prefer aspect-ratio preservation/padding when scientifically appropriate, as required by `PLAN.md`.
- Validate positive target sizes and original image/mask size agreement. Keep metadata keys stable: `original_size`, `resized_size`, `padding`, and `target_size`.

## Dataset facts and split policy

The current local scan found:

- 4,413 input image files under `water_v2/JPEGImages` across 22 top-level source groups.
- 2,400 image-like annotation files and 2,400 discoverable image/mask pairs.
- `train.txt` contains only `ADE20K` and `river_segs`, which provide 2,188 paired samples.
- `val.txt` lists the other 20 groups, which provide 212 paired samples.
- The U-Net notebook splits those 20 evaluation groups with `GroupShuffleSplit(test_size=0.5, random_state=42)`, yielding 83 validation images from 10 groups and 129 test images from 10 groups.
- The current `IMGs/` folder contains 10 original JPEGs plus `img9_blurred.png`, so `list_local_images()` currently returns 11 supported images. Stored notebook/README statements reporting 10 predate the blurred derivative.
- `MP4/` contains six videos.

Many Water-v2 source folders contain more unannotated input images than masks. Never assume every file under `JPEGImages` is trainable; use the pairing helper.

Group identity is the first directory component below `JPEGImages`. Images from the same scene/video/source group must not cross train, validation, or test partitions. A random image-level split can leak near-duplicate scene content and is not acceptable for final model claims.

The present official split is itself a known weakness: only two broad groups are used for training while 20 groups are reserved for evaluation. `PLAN.md` calls for auditing group meaning and creating a more balanced group-aware split, approximately 14-16 train groups, 3-4 validation groups, and 3-4 final-test groups, or using `GroupKFold`/`StratifiedGroupKFold` when appropriate.

Keep the final test set untouched until model architecture, loss, preprocessing, and threshold are frozen. For real washery video, keep all frames from one video/camera sequence in the same partition.

## Implemented helpers and semantics

### Baseline (`baselines.py`)

`blue_dominance_threshold()` expects normalized RGB HWC input. A pixel is water when all of the following hold:

- blue >= `blue_min` (default 0.18)
- blue >= red * `blue_red_ratio` (default 1.05)
- blue >= green * `blue_green_ratio` (default 0.85)
- mean RGB brightness >= `brightness_min` (default 0.05)

It returns a binary `uint8` mask. `mask_coverage()` returns the percentage of positive pixels. This method is intentionally explainable and fragile; it is a baseline, not the target solution.

### Metrics (`metrics.py`)

- `binarize()` uses `>= threshold` and rejects thresholds outside `[0, 1]`.
- `intersection_over_union()` measures binary water-class IoU.
- `dice_coefficient()` measures binary water overlap.
- `pixel_accuracy()` measures all pixels and can be misleading when background dominates.
- IoU and Dice use an epsilon in numerator and denominator, so two empty masks score approximately 1.0.

Report water IoU and Dice as primary metrics, with precision, recall, per-image metrics, per-group metrics, confusion counts, and qualitative error views. Accuracy alone is never sufficient. The roadmap also requires Boundary F1 for contour quality.

### Visualization (`visualization.py`)

- Visualization helpers expect RGB images and binary/positive masks with matching spatial shapes.
- `overlay_mask()` uses blue `(0.0, 0.45, 1.0)` and alpha `0.45` by default, returns a clipped float image, and treats every positive mask value as selected.
- `show_image_mask_overlay()` is the standard three-panel view: image, mask, overlay.

For model comparisons, prefer the six diagnostic views described in `PLAN.md`: original, ground truth, probability map, binary prediction, error overlay, and distinct false-positive/false-negative rendering.

### Overflow rules (`overflow.py`)

- `rectangular_zone()` converts relative coordinates to a boolean zone mask.
- `water_ratio_in_zone()` computes the fraction of critical-zone pixels marked as water and returns `0.0` for an empty zone.
- `overflow_confidence()` linearly maps zone coverage to `[0, 1]`, saturating at `alert_ratio` (default 0.15).
- `overflow_label()` maps confidence to `normal` below 0.3, `surveillance` from 0.3 to below 0.7, and `debordement_probable` from 0.7 upward.

The confidence is explicitly not a calibrated probability. Coordinates, zone shape, `alert_ratio`, and label thresholds must eventually be calibrated per fixed camera with real annotated events. New code should validate coordinate ranges, compatible mask shapes, and threshold ordering because the current helpers perform limited validation.

## Experiment record

Treat notebook outputs as historical evidence, not automatically reproducible benchmark claims. Record the exact code revision, split, seed, hardware, preprocessing, threshold, and checkpoint for new results.

### Exploration notebook

- Stored output reports 2,400 paired Water-v2 samples and 10 local images at the time it was run.
- Examples are directly resized to `(384, 256)` for display.

### Threshold baseline notebook

- Runs the default blue-dominance rule on only the first paired sample at `(384, 256)`.
- Stored single-image result: IoU `0.6007723187`, Dice `0.7506030829`.
- This is not a dataset-level baseline and must not be compared as if it were.

### Classical ML notebook

- Features: per-pixel RGB plus OpenCV HSV (six columns).
- Direct resize: `(384, 256)`.
- Current notebook randomly splits 2,400 images into 1,920 train and 480 test images; it is image-disjoint but not group-aware.
- It samples up to 50,000 pixels per image, materializing 96 million train rows and 24 million test rows, then fits on a seeded 250,000-pixel train subsample.
- Models: balanced Logistic Regression, Linear SVM, balanced Decision Tree, and balanced-subsample Random Forest.
- Stored best result is Random Forest: accuracy `0.669971`, water precision `0.400401`, recall `0.545470`, F1/Dice `0.461811`, IoU `0.300230`.
- The notebook can consume substantial RAM because the full pandas train/test matrices are built before fit-time subsampling. Redesign sampling/streaming before rerunning on constrained hardware.
- Do not use these results as final generalization results until the split is group-aware.

### U-Net notebook and checkpoint

- Framework: PyTorch with mandatory CUDA; the notebook raises if CUDA is unavailable to prevent accidental long CPU training.
- Stored run environment: PyTorch `2.13.0+cu126`, CUDA runtime `12.6`, GTX 1650 Max-Q with 4 GB VRAM.
- Seed: 42 for Python, NumPy, PyTorch, CUDA, loader generator, and group split.
- Input: direct resize to `256 x 256`, RGB `[0,1]`, binary mask.
- Batch size 4, `num_workers=0` for Windows/Jupyter, pinned memory, mixed FP16 precision.
- Training augmentations: 50% horizontal flip and 20% vertical flip. The roadmap warns against unrealistic vertical transformations, so review the vertical flip before future controlled experiments.
- Architecture: four-level custom U-Net, `BASE_FILTERS=16`, BatchNorm+ReLU double convolutions, transposed-convolution decoder, one logit output channel, 1,942,577 parameters.
- Loss: `0.5 * BCEWithLogitsLoss + 0.5 * DiceLoss`.
- Optimizer: AdamW, learning rate `3e-4`, weight decay `1e-4`.
- Scheduler: ReduceLROnPlateau on validation IoU, factor `0.5`, patience 2.
- Maximum 30 epochs, early-stopping patience 6, prediction threshold 0.5.
- Selection: best mean per-image validation IoU.
- Training stopped after epoch 27. The best saved checkpoint is epoch 21.
- Checkpoint metadata: image size `(256, 256)`, base filters 16, threshold 0.5.
- Stored checkpoint validation metrics: loss `0.3310664`, Dice `0.7323694`, IoU `0.6287569`, accuracy `0.8610038`.
- Stored notebook test metrics: loss `0.3240`, Dice `0.7625`, IoU `0.6517`, accuracy `0.8498`.

The U-Net classes and inference preprocessing live only in notebook 03. To build inference, first move an architecture with checkpoint-compatible names/shapes into the package, load the checkpoint with the recorded config, use identical preprocessing, apply sigmoid then the stored threshold, and restore predictions to original geometry. Verify numerical agreement with notebook outputs before refactoring further.

PyTorch checkpoint loading can execute pickle content. Only load trusted checkpoints. Use `map_location` deliberately and do not assume CUDA is available for metadata inspection.

### Overflow notebook

- Loads a local image at `(384, 256)`.
- Uses an all-zero prediction placeholder.
- Defines the bottom quarter as a rectangular critical zone.
- Uses `alert_ratio=0.15` and the default three-level label thresholds.
- It proves only the geometry/scoring API wiring; it does not validate water or overflow detection.

## Research priorities from `PLAN.md`

Work in this order unless the user explicitly changes scope:

1. Audit Water-v2 pairing, alignment, mask values, empty masks, duplicates, water ratios, resolutions, aspect ratios, group meaning, and annotation quality.
2. Replace the current imbalanced split with a reproducible group-aware protocol and preserve a final untouched test set.
3. Re-evaluate the current U-Net with water IoU, Dice, precision, recall, Boundary F1, per-image and per-group metrics, and qualitative errors.
4. Establish pretrained-encoder U-Net baselines (ResNet18 and/or MobileNetV3).
5. Compare precision-oriented candidates under the same protocol: U-Net++ with a light backbone, DeepLabV3+ with MobileNetV3/ResNet18, Attention U-Net, and SegFormer-B0.
6. Optimize only the best candidates: loss, learning rate, resolution/patches, threshold, boundary term if justified, and at least three seeds.
7. Produce a reliable single-image segmentation script and full experiment artifacts.
8. Extract diverse frames from the six real videos, annotate approximately 100-200, measure domain shift, fine-tune on real data, and validate by whole video.
9. Only after reliable real-domain masks, add per-camera polygons/level lines/danger zones, temporal hysteresis, camera-health checks, alerts, and evidence storage.
10. Optimize for deployment later with ONNX/ONNX Runtime/TensorRT and validate output parity before performance claims.

The target hardware has 4 GB VRAM (GTX 1650 Max-Q and RTX 3050 Laptop). Favor light backbones, AMP, batch sizes 1-2 when required, gradient accumulation, and reproducible memory measurements. Quality is currently more important than real-time speed. The future target is roughly 5-10 FPS per camera with sub-second decision latency, subject to site validation.

## Coding and experiment conventions

- Use Python 3.12-compatible syntax and libraries already declared in `pyproject.toml` where practical.
- Follow existing PEP 8-style naming: snake_case functions/variables, PascalCase classes, uppercase module constants.
- Prefer type hints on reusable public helpers, but do not add unused imports or type noise.
- Write concise docstrings that state array shapes, dtypes, ranges, units, interpolation, and threshold semantics.
- Existing user-facing documentation/UI is mostly French; shared low-level docstrings are mostly English. Match the surrounding file unless a task requests a language cleanup.
- Avoid duplicating helper functions in notebooks. Refactor reusable code into `src/` and import it.
- Use deterministic seeds and log them. Log complete configs, split/group names, best epoch, threshold, runtime, and maximum VRAM for training runs.
- Compare models on the same split, preprocessing, augmentations, threshold-selection rules, and metrics.
- Select checkpoints using validation water IoU, not accuracy. Never select or tune using final-test results.
- Preserve original aspect ratio when possible; if using letterboxing, exclude padding from metrics or restore the prediction before evaluation.
- Avoid large, unrealistic rotations, vertical transformations, extreme color shifts, or any transform that changes mask boundaries independently of the image.
- Save generated artifacts under a clearly named output/log directory rather than alongside source. Check Git status before finishing.
- Do not hand-edit `uv.lock`, compiled requirement pins, notebook binary attachments, `.pt` files, or media.
- When dependencies change, update `pyproject.toml`, refresh `uv.lock`, and regenerate/verify `requirements.txt` intentionally.

## Validation expectations

There is no automated test suite today. Validation must be proportional to the change, and new reusable behavior should normally receive focused tests when a test framework is introduced.

Minimum checks for Python source changes:

```powershell
.\.venv\Scripts\python.exe -m compileall -q src apps main.py
```

Also run focused smoke checks for affected contracts:

- image loaders return RGB `float32` arrays in `[0,1]`;
- masks remain 2-D `uint8` arrays with values `{0,1}`;
- image/mask spatial dimensions match after resize or padding;
- restored masks match original geometry;
- metric functions cover perfect overlap, disjoint masks, and empty masks;
- overflow zones and water masks have compatible shapes;
- GUI changes are exercised interactively on at least one image;
- notebook refactors are checked on a small sample before expensive full execution;
- checkpoint refactors produce the same output as the notebook for a fixed sample within an explicit numerical tolerance.

Do not launch the full U-Net training notebook as a routine validation step. It requires CUDA and is expensive. Do not launch GUI applications in headless environments. If CUDA is unavailable, say which checks were skipped rather than weakening the notebook's guard or silently training on CPU.

For experiment changes, report global and per-group water IoU/Dice, precision, recall, Boundary F1 when available, confusion counts, mean and distribution of per-image scores, best/worst qualitative samples, runtime, and VRAM. Verify that masks use nearest-neighbor interpolation and that no group leakage occurred.

## Known pitfalls and technical debt

- `main.py` is not the application entry point.
- The tracked U-Net checkpoint cannot yet be consumed through a package API because the architecture is notebook-local.
- The GUI does not use the U-Net.
- The local overflow notebook does not perform inference and cannot substantiate overflow performance.
- `train_val_split()` is deterministic but item-random, not group-aware; do not use it for final scene/video evaluation.
- The classical ML notebook's split is also not group-aware and its full DataFrames are very large.
- Direct resizing in existing notebooks changes aspect ratios.
- The U-Net notebook's vertical-flip augmentation conflicts with the roadmap's caution about unrealistic transforms.
- `requirements.txt` and `pyproject.toml` currently disagree on direct runnable dependencies.
- Notebook outputs can be stale relative to current files, environments, and data counts.
- Several notebooks contain large embedded images/outputs and mixed text encodings. Avoid needless output churn or blanket reformatting.
- Metrics average per-batch, per-image scores in notebook 03; clearly distinguish that from global pixel aggregation in future comparisons.
- Empty-mask IoU/Dice behavior is defined by smoothing; state the convention when comparing with other libraries.
- Overflow helpers do not fully validate coordinate bounds, mask shapes, `alert_ratio`, or label-threshold ordering.
- `outputs/` is not ignored even though the GUI creates it.
- Raw industrial images/videos may be sensitive operational data. Keep them local and avoid reproducing identifying content in logs, reports, or commits without approval.
- Water-v2 performance does not establish washery-domain performance. Expect substantial domain shift from muddy water, foam, dust, reflections, fixed camera angles, compression, people, and equipment.
